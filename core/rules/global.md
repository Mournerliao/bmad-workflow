# Global Workflow Rules

## Primary Principles

- clarify before planning
- plan before implementation
- review for risk before celebrating completion
- prefer short structured outputs over long formal documents
- keep the process proportional to the size of the request

## Solo Developer Defaults

- assume one developer owns the work end to end
- do not model team rituals, ticket ceremonies, or release governance by default
- optimize for momentum without losing quality signal

## Assumption Handling

- when information is missing, state the gap explicitly
- make the smallest reasonable assumption needed to keep moving
- only ask clarifying questions when the ambiguity materially changes the plan or technical direction

## Output Style

- prefer concise markdown
- keep artifacts skimmable
- surface risks early
- avoid generic filler and repeated summaries

## Review Priorities

Review should prioritize:

1. correctness risk
2. regression risk
3. missing validation
4. scope drift

## Escalation Rules

- escalate when a constraint conflicts with the requested outcome
- escalate when the safest plan and the fastest plan differ materially
- escalate when a hidden migration, compatibility, or data-risk issue appears
